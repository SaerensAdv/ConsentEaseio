ConsentEase is being repositioned as an agent-native consent layer: a versioned public REST API (`/api/v1`) authenticated by API keys, plus an MCP server that is a thin client over that API.

- **Strategy doc:** `docs/concept/consentease-connect-concept.md`.
- **Build sequence (tickets):** `docs/plans/connect-phase1-build-sequence.md`.
- **Security reference (authoritative for this surface):** `threat_model.md` — covers API-key auth, REST, and MCP.

Key durable decisions (details in threat_model.md / the build plan):
- API key format `ce_live_<keyId>_<secret>`; only `keyId` (lookup) + `secretHash = HMAC-SHA256(API_KEY_PEPPER, secret)` are stored — never the plaintext secret.
- `API_KEY_PEPPER` MUST be distinct from `SESSION_SECRET` (crypto fails closed if they match).
- Idempotent billable writes de-dupe on `api_usage_events (apiKeyId, idempotencyKey)`; NULL idempotencyKeys are distinct in Postgres so non-idempotent events never collide.
- API keys inherit exactly the owner's dashboard website access (agencies include in-account client sites) — never broader. Every resource access must pass an explicit authorization check; authentication alone grants nothing.
- `secretHash` is credential-verifier material: never serialize a raw `ApiKey` row to a client. List goes through a `secretHash`-free projection; create/other handlers must strip it via `toPublicApiKey()`. Only the auth lookup (`getApiKeyByKeyId`) may read `secretHash`.
- `recordApiUsageEvent` returns `conflict: true` when an idempotency key is reused for a *different* `action`; the caller (route) must reject that as 409 rather than treat it as a replay. (Full request-body hashing is a deliberate future enhancement, not done yet.)
- `/api/v1` rate limiting is **two-stage**: a cheap per-IP limiter (`apiV1IpLimiter`) MUST be mounted *before* `apiKeyAuth`, and the per-key tier limiter (`apiV1RateLimiter`) *after* it. **Why:** a single post-auth per-key limiter leaves the unauthenticated surface (the `getApiKeyByKeyId` DB lookup) open to well-formed-random-key floods; a pre-auth limiter can only key on IP. The IP ceiling is deliberately set above the highest per-key tier so a legitimate high-tier client behind one IP is never throttled at stage 1.
- Rate-limit IP keying MUST come from Express `req.ip` (`app.set('trust proxy', 1)` is set in `server/index.ts`), NEVER from parsing raw `X-Forwarded-For`. **Why:** the upstream proxy *appends* the real client IP, so the leftmost XFF value is attacker-controlled; keying on `XFF[0]` lets a caller rotate spoofed IPs and bypass every IP-keyed limiter (auth brute-force guard, public-ingest guard, and the pre-auth /api/v1 key-lookup guard). This is project-wide, not just /api/v1.
- Idempotency is **consent-only** in Phase 1 (`POST /api/v1/consent` is the sole billable write, 1 unit). The other POSTs (`/sites`, `/sites/:id/scan`, `/sites/:id/cookies`) are intentionally NON-idempotent. **Why:** their meter runs *after* the resource is created, so a reused-key conflict would return a 409 *after* a successful create — dishonest. Full create-replay caching (store + return the original resource) is deliberate future work.
- Consent idempotency resolves the `api_usage_events` pre-check (`getApiUsageEventByIdempotency`) BEFORE writing the consent record, returning `200 {deduped:true}` on replay without re-writing; only on a miss does it `createConsentLog` THEN `meterApiUsage(1)`. **Why:** meter-first would risk losing a legal consent record if metering failed. Known accepted Phase-1 gap: two truly-concurrent identical requests can both pass the pre-check and double-write the consent row (the unique `(apiKeyId, idempotencyKey)` ledger index still de-dupes billing). Low-cost hardening (in-process keyed mutex now, Postgres advisory lock later) is deferred, not blocking.
- The `:id` path param on every `/api/v1` resource route is Zod-validated once via a shared `apiV1Router.param("id", ...)` guard (→ 400 on malformed) so it never reaches `storage` unchecked, satisfying the threat-model "all params validated" rule without repeating the check per route.
- API-key ROTATION must enforce the same simultaneously-valid-key cap as creation (currently 25), counted INSIDE the rotation transaction. **Why:** rotation mints a successor while the predecessor stays valid through its grace window, so it transiently +1's the valid-key count; without a cap there, a user can rotate the newest active key over and over and stack unbounded still-valid grace keys, bypassing the create-path ceiling (credential sprawl). **How to apply:** count non-revoked + non-expired keys for the owner and reject (403) when adding the successor would exceed the limit; `rotateApiKey` returns a discriminated result (`cap_exceeded` vs `not_found`) so the route can map 403 vs 404.
- The MCP server lives in `mcp-server/` as a SELF-CONTAINED npm package (own `package.json` + `node_modules`), isolated from the root `tsconfig` (root `include` is only client/shared/server) so it never enters `npm run check`. It is a THIN stdio client over `/api/v1`, configured solely by env (`CONSENTEASE_API_KEY` required, `CONSENTEASE_API_URL` origin, default localhost:5000), and inherits exactly the key's access/scopes — no extra privilege.
- MCP exposes 9 tools, all prefixed `consentease_`, each mapping 1:1 to a real `/api/v1` endpoint. **The plan's `enable_consent`/`configure_banner` tools were intentionally DROPPED** — they have no backing endpoint. **Why:** a thin client can only expose what the API offers, and `create_site` already auto-provisions the banner config + default cookie categories + a scan, so the "no-consent → scanned + consent-enabled + embed" flow is `create_site` → (`scan_site`) → `get_embed_snippet`. Compliance output is operational status only — explicitly NOT legal advice.

**Why:** This surface is security-critical and spans many sessions; the rules above are easy to violate by accident.
**How to apply:** Before touching API-key auth, `/api/v1`, or the MCP server, re-read `threat_model.md` and the build-sequence plan rather than re-deriving the design.
